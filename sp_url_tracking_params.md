# Техническое задание: добавление GET-параметров TT и TD в URL

## 1. Цель задачи

### 1.1. Бизнес-проблема

Нужно сохранять единый Track ID пользователя в ситуациях, когда стандартные браузерные механизмы (cookies, localStorage) не работают или не переносятся. Track ID привязан к браузеру, но в идеальном сценарии должен оставаться одинаковым у одного и того же пользователя в разных браузерах — поэтому его нужно уметь переносить между ними:

1. **⚠️ Возврат после внешней авторизации (критичный сценарий).** Пользователь покидает сайт ради OAuth-логина через **Google, Discord, Apple, Facebook** и подобные внешние провайдеры, после чего возвращается обратно. В ряде случаев (особенно при возврате из inapp-WebView, при работе через сторонние домены провайдера, при заблокированных third-party cookies или Safari ITP) браузерный контекст возврата может отличаться от исходного — и Track ID теряется ровно в тот момент, когда он наиболее важен (момент конверсии в зарегистрированного пользователя). **Этот сценарий должен поддерживаться в первую очередь.**
2. **Кросс-браузерный переход.** Пользователь открыл сайт во встроенном браузере приложения (inapp WebView в Instagram, TikTok, Facebook и т.п.) и затем нажал «Открыть в Safari/Chrome». Системный браузер не имеет доступа к cookies и storage inapp-браузера — Track ID теряется, тот же пользователь выглядит как новый.
3. **Заблокированные cookies / storage.** Safari ITP, режим инкогнито, корпоративные политики, AdBlock-расширения, отказ пользователя от cookies в баннере согласия — всё это приводит к тому, что Track ID невозможно сохранить или прочитать обычным способом.
4. **Переходы между страницами.** Идентификатор должен переживать любую внутреннюю навигацию (включая полную перезагрузку и SPA-роутинг), не теряясь между страницами.

### 1.2. Ограничение: защита от ложного склеивания

Если пользователь скопировал ссылку и поделился ею (мессенджер, соцсеть, email), то получатель ссылки **не должен** унаследовать Track ID отправителя — иначе действия разных людей сольются под одним идентификатором.

Поэтому механизм передачи Track ID должен иметь признак «свежести»: идентификатор переносится между контекстами только в том случае, если переход произошёл вскоре после генерации ссылки (порядок единиц–десятков минут). Старые ссылки должны приводить к генерации нового Track ID у получателя.

### 1.3. Рекомендуемое решение

В разделах ниже описан рекомендуемый подход — передача Track ID и timestamp через GET-параметры `td` и `tt` в URL:

- `td` — Track ID (тот же, что уже генерируется на сайте);
- `tt` — timestamp последнего обновления URL, используется для проверки «свежести» при попытке унаследовать `td` из URL.

Это решение выбрано как наиболее простое и независимое от хранилищ браузера: URL переносится между браузерами автоматически (через «Открыть в…», копирование адресной строки, share-меню системы) и недоступен для блокировки расширениями, в отличие от cookies/storage.

**Альтернативные подходы допустимы**, если они закрывают сценарии 1.1.1–1.1.4 и удовлетворяют ограничению 1.2 — например, server-side линковка по fingerprint, Storage Access API, postMessage между inapp и системным браузером и т.п. В таком случае альтернативу нужно согласовать до начала реализации.

## 2. Параметры

### 2.1. `tt` — timestamp

Параметр `tt` должен содержать timestamp момента генерации метки.

Рекомендуемый формат:

```text
tt=1714912345678
```

Где значение — Unix timestamp в миллисекундах, полученный через:

```javascript
Date.now()
```

### 2.2. `td` — track ID

Параметр `td` должен содержать уникальный идентификатор трекинга.

Это тот же Track ID, который уже генерируется на странице сайта — новый идентификатор создавать не нужно, используется существующий.

## 3. Основная логика работы

### 3.1. При первом входе на сайт

При открытии любой страницы сайта фронтенд должен проверить наличие параметров `tt` и `td` в текущем URL.

Если параметры отсутствуют, необходимо:

1. Сгенерировать `tt`.
2. Сгенерировать `td`.
3. Добавить оба параметра в текущий URL без перезагрузки страницы.

Пример:

Исходный URL:

```text
https://example.com/page?utm_source=facebook
```

После обработки:

```text
https://example.com/page?utm_source=facebook&tt=1714912345678&td=7f3a9c2b8d1e4f60a91c3b5d2e8f9a0b
```

Добавление параметров должно выполняться через History API, например `history.replaceState()`, чтобы не вызывать перезагрузку страницы.

### 3.2. Если параметры уже есть в URL

Логика обработки пришедших в URL параметров `tt` и `td` зависит от того, есть ли у пользователя уже сохранённый на сайте Track ID.

#### 3.2.1. У пользователя уже есть Track ID

Если у пользователя уже существует собственный Track ID, пришедшие в URL значения `tt` и `td` игнорируются.

Параметры в URL заменяются актуальными значениями пользователя — так, как будто `tt` и `td` в URL не приходили вовсе.

#### 3.2.2. У пользователя нет Track ID

Если собственного Track ID у пользователя ещё нет, выполняется попытка перенять значение из параметра `td`:

1. Вычисляется разница между текущим временем (`Date.now()`) и значением параметра `tt`.
2. Если разница составляет не более 10 минут (600 000 мс), значение параметра `td` рассматривается как кандидат на Track ID пользователя.
3. Перед использованием значение `td` обязательно проходит валидацию формата по тем же правилам, которые применяются при стандартной генерации Track ID на сайте. Если формат не соответствует ожидаемому — значение `td` отбрасывается.
4. Если разница по `tt` превышает 10 минут, либо `td` не прошёл валидацию формата — Track ID генерируется заново стандартным способом, принятым на сайте, а `tt` устанавливается в `Date.now()`.

#### 3.2.3. Запись итоговых значений в URL

После того как значения `tt` и `td` определены по правилам выше, они записываются в URL через `history.replaceState()` — в том числе поверх тех значений, с которыми пользователь пришёл.

Пример (у пользователя уже есть Track ID `existing-track-id`, в URL пришли `tt=1714912345678&td=abc123`):

```text
До:    https://example.com/page?tt=1714912345678&td=abc123
После: https://example.com/page?tt=<Date.now()>&td=existing-track-id
```

## 4. Поведение при переходах между страницами

После любого перехода пользователя на другую страницу сайта параметры `tt` и `td` должны сохраняться в URL.

Это касается:

- переходов по внутренним ссылкам;
- переходов через меню;
- переходов через кнопки;
- SPA-навигации, если сайт работает как Single Page Application;
- программных переходов через JavaScript.

При каждом изменении значения URL (любой переход, обновление route/path в SPA, программный `pushState`/`replaceState`) параметр `tt` должен перезаписываться актуальным `Date.now()` — он всегда отражает время последнего изменения URL и должен оставаться максимально свежим. Параметр `td` при этом сохраняет своё значение.

Пример:

Пользователь находится на странице:

```text
https://example.com/page-a?tt=1714912345678&td=abc123
```

Переходит на:

```text
https://example.com/page-b
```

Фактический URL после перехода должен быть (где `<Date.now()>` — момент перехода):

```text
https://example.com/page-b?tt=<Date.now()>&td=abc123
```

## 5. Важные ограничения

- Нельзя удалять существующие GET-параметры, например `utm_source`, `gclid`, `fbclid` и другие.
- Нельзя перезагружать страницу только ради добавления параметров.
- Нельзя добавлять параметры к внешним ссылкам, ведущим на другие домены.

## 6. Требования к внутренним ссылкам

Реализовать глобальный перехватчик навигации, который при любом переходе по внутренней ссылке автоматически добавляет `tt` и `td` к целевому URL перед переходом. Ссылки в шаблонах параметров не содержат.

Перехватчик должен охватывать оба типа переходов:

- **SPA-переходы** — через router guard (Vue Router `beforeEach`, React Router и т.п.);
- **Не-SPA-переходы** (полная перезагрузка страницы) — через глобальный обработчик кликов на `document`, перехватывающий все `<a href="...">` с внутренними адресами.

Параметры, добавляемые при переходе:

- `tt` — `Date.now()` в момент клика/навигации (всегда свежий, задаётся в момент перехода);
- `td` — текущий Track ID пользователя (сохраняет своё значение).

Пример:

Исходная ссылка в шаблоне:

```html
<a href="/checkout">Checkout</a>
```

При клике пользователь попадает на:

```text
/checkout?tt=<Date.now()>&td=abc123
```

Если ссылка уже содержит параметры:

```text
/checkout?step=1
```

Итоговый URL должен быть:

```text
/checkout?step=1&tt=<Date.now()>&td=abc123
```

## 7. Требования для SPA

Если сайт использует SPA-навигацию, необходимо обрабатывать изменения URL внутри приложения.

После каждого изменения route/path необходимо проверять наличие `tt` и `td` в URL и добавлять их при отсутствии.

Обработка должна работать для:

- `pushState`;
- `replaceState`;
- `popstate`;
- роутера приложения, если используется React Router, Vue Router, Next.js Router или аналог.

## 8. Критерии приёмки

Задача считается выполненной, если:

1. При первом открытии сайта без `tt` и `td` оба параметра автоматически появляются в URL.
2. Добавление параметров происходит без перезагрузки страницы.
3. Существующие GET-параметры не удаляются и не изменяются.
4. Если `tt` и `td` уже есть в URL, значение `td` не перегенерируется. Значение `tt` всегда обновляется до текущего `Date.now()`.
5. При переходе на другую внутреннюю страницу `tt` и `td` сохраняются.
6. Параметры не добавляются к внешним ссылкам.
7. Логика работает для всех основных внутренних переходов на сайте.
8. Значение `td` не содержит персональных данных.
9. Решение протестировано на desktop и mobile браузерах.
10. Решение протестировано для обычной навигации и SPA-навигации, если она используется на сайте.

## 9. Примеры тест-кейсов

### Тест 1: первый вход без параметров

Открыть:

```text
https://example.com/
```

Ожидаемый результат:

```text
https://example.com/?tt=<timestamp>&td=<track_id>
```

### Тест 2: вход с существующими UTM-параметрами

Открыть:

```text
https://example.com/?utm_source=facebook&utm_campaign=test
```

Ожидаемый результат:

```text
https://example.com/?utm_source=facebook&utm_campaign=test&tt=<timestamp>&td=<track_id>
```

### Тест 3: вход с уже существующими `tt` и `td`

Открыть (у пользователя уже есть Track ID `abc`):

```text
https://example.com/?tt=111&td=abc
```

Ожидаемый результат:

```text
https://example.com/?tt=<Date.now()>&td=abc
```

Значение `td` сохраняется, `tt` обновляется до текущего момента.

### Тест 4: внутренний переход

Открыть:

```text
https://example.com/page-a?tt=111&td=abc
```

Перейти на:

```text
/page-b
```

Ожидаемый результат (где `<Date.now()>` — момент перехода):

```text
https://example.com/page-b?tt=<Date.now()>&td=abc
```

### Тест 5: внешняя ссылка

Если на странице есть ссылка:

```text
https://external-site.com/
```

Параметры `tt` и `td` не должны добавляться к этой ссылке.
